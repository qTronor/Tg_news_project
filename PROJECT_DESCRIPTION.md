# Telegram News Intelligence Platform — Полное описание проекта

## Для чего этот проект

Система мониторинга и анализа новостей из Telegram-каналов. Она автоматически собирает сообщения, обогащает их несколькими слоями ML-анализа и превращает сырой поток текстов в структурированную аналитику: кто о чём пишет, как меняется тональность, какие персоны и организации упоминаются, как темы распространяются между каналами и насколько каждая из них важна прямо сейчас.

**Ключевые вопросы, на которые отвечает система:**
- Что происходит в медиапространстве прямо сейчас?
- Как менялась повестка за последние часы/дни?
- Кто первым написал о теме, и как она распространяется?
- Насколько важна тема — растёт ли её охват, появляются ли новые участники?

Побочные сервисы:

**Почему такая топология, а не монолит:**

Монолит проще запустить, но при росте нагрузки его невозможно масштабировать частично. Здесь sentiment_analyzer — самый медленный сервис (GPU inference), и его можно запустить в 4 экземплярах, не трогая остальные. Каждый сервис также деплоится и обновляется независимо: смена ML-модели в NER не требует остановки collector'а.

**Почему Kafka, а не прямые вызовы между сервисами:**

При прямых HTTP-вызовах failure одного сервиса каскадирует на всю цепочку. Kafka создаёт буфер: collector может продолжать собирать данные, пока preprocessor перезапускается. Сообщения хранятся 30 дней — можно переиграть историю через любой сервис без потери данных.

## Инфраструктура

### PostgreSQL — основное хранилище

**Почему не MongoDB/Cassandra:**
- Новости — реляционные данные: сообщение → препроцессинг → sentiment → NER → кластер. JOIN'ы — это именно то, что нужно для аналитики.
- ACID-транзакции критичны для идемпотентности: `ON CONFLICT DO NOTHING` — это одна атомарная операция в PostgreSQL, в MongoDB пришлось бы эмулировать через findAndModify.
- `postgres:15-alpine` — минимальный образ, production-ready.

**Две базы данных:**
- `telegram_news` — данные пайплайна (сообщения, NER, sentiment, кластеры)
- `tg_news_auth` — аутентификация, изолирована намеренно: проблема в auth не должна затронуть доступ к аналитическим данным

### Neo4j — граф связей

**Почему не хранить граф в PostgreSQL:**
Запрос «найди все сущности, связанные с Газпромом через 2 прыжка» на PostgreSQL — это самосвязанный рекурсивный CTE. На Neo4j — одна строка Cypher. Для анализа пропагации новостей (кто на кого ссылается) и co-occurrence сетей графовая БД — правильный инструмент.

**Почему Neo4j 5.15-community, а не 5.x enterprise:**
Community-версия бесплатна и поддерживает все нужные операции. Enterprise нужен только при кластеризации самого Neo4j (несколько узлов), что избыточно для данного проекта.

### Apache Kafka — транспорт

**Почему не RabbitMQ или Redis Pub/Sub:**
- Kafka хранит все сообщения до истечения retention-периода (raw: 30 дней, DLQ: 90 дней). RabbitMQ удаляет сообщение после подтверждения потребителем. Это значит, что при добавлении нового сервиса (например, graph-builder) его можно «прочитать историю» с начала, не трогая другие сервисы.
- 6 партиций на топик позволяют 6 параллельным потребителям обрабатывать сообщения одновременно.
- LZ4-компрессия уменьшает сетевой трафик примерно в 3× для JSON-событий.

**Почему 6 партиций, а не 1 или 12:**
Партиции — единица параллелизма. 6 соответствует типичной конфигурации dev-кластера с запасом для горизонтального масштабирования. 12 избыточны при одном брокере (overhead на репликацию метаданных).

### Redis — кеш идемпотентности

`allkeys-lru` + 512MB: при нехватке памяти Redis вытесняет наименее используемые ключи. Это нормально для идемпотентности — сообщения старше N дней всё равно не придут повторно. Альтернатива — хранить все `event_id` в PostgreSQL (таблица `processed_events`, 7-дневная ретенция), что и сделано как fallback.

## Сервисы: описание, логика, обоснования

### 1. rbc_telegram_collector

**Что делает:** Собирает сообщения из публичных Telegram-каналов.

**Почему Telethon + MTProto, а не Telegram Bot API:**
Bot API ограничен публичными каналами где бот является участником, и даёт только новые сообщения (нет истории). MTProto — нативный протокол Telegram, позволяет читать историю любого публичного канала без ограничений, с пагинацией. `TG_STRING_SESSION` позволяет авторизоваться один раз и переиспользовать сессию.

**Режимы запуска:**
- `service` — долгоживущий процесс, пересобирает каналы каждые 60 секунд
- `one-shot` — разовый запуск для backfill

**Backfill:** При добавлении нового канала сервис автоматически забирает его историю. `state/` директория хранит позицию последнего собранного сообщения — перезапуск не приводит к дублированию.

**Ключ партиционирования Kafka = `event_id`:**
Все сообщения одного канала попадают в одну партицию (key-based routing), что гарантирует порядок обработки в рамках канала.

### 2. message_persister

**Что делает:** Kafka → PostgreSQL, таблица `raw_messages`.

**Почему это отдельный сервис, а не часть collector'а:**
Разделение ответственности. Collector работает с внешним API (Telegram), persister работает с базой. Если база недоступна, collector продолжает писать в Kafka, persister наверстает после восстановления. Если объединить — при проблеме с БД останавливается весь сбор.

**Идемпотентность:**
```sql
INSERT INTO raw_messages ... ON CONFLICT (channel, message_id) DO NOTHING
```
Kafka at-least-once delivery гарантирует доставку, но может доставить дважды. `ON CONFLICT` — единственная защита на уровне БД.

**DLQ-логика:** Если текст пустой или JSON невалиден — немедленно в DLQ (не retriable). Если БД недоступна — экспоненциальный backoff (1s → 2s → 4s → DLQ). Различение retriable/non-retriable важно: бесконечный retry невалидного сообщения заблокирует всю партицию.

### 3. preprocessor

**Что делает:** Очищает текст, определяет язык, извлекает структуру (URL, @mentions, #hashtags), вычисляет fingerprints.

**Почему fasttext для определения языка, а не langdetect:**
fasttext (`lid.176.ftz`) — это скомпилированная модель на 176 языков с инференсом < 1ms. langdetect основан на n-gram статистике и ненадёжен на коротких текстах (менее 50 слов). Telegram-посты часто короткие. fasttext даёт уверенный ответ с confidence score — всё, что ниже 0.5, помечается как неопределённое.

**Simhash64 — зачем:**
Двойные новости — норма для телеграм-каналов: один и тот же пост рерайтится десятками каналов. SHA-256 хеш поймает точные копии. Simhash64 ловит почти-дубликаты (перефразированные версии): Hamming distance ≤ 3 означает ~95% похожих текстов. Это не фильтрует дубликаты автоматически, но флажок помогает при анализе.

**URL fingerprint (SHA-256):**
Одна ссылка, распространённая по 50 каналам, даёт одинаковый fingerprint. source_resolver использует это, чтобы найти первый источник новости: самый ранний документ с этим fingerprint.

**`analysis_mode` — ключевое поле пайплайна:**
- `full` → русский или английский → проходит весь пайплайн (sentiment + NER + кластеризация)
- `partial` → другой язык → только кластеризация (мультиязычные embeddings работают для любого языка)

Это решение — компромисс: полноценные NER-модели для 30+ языков потребовали бы 50+ GB моделей. Использование двух лучших специализированных моделей (Natasha для RU, BERT для EN) даёт более высокое качество для основного контента.

### 4. sentiment_analyzer

**Что делает:** Определяет тональность (positive/negative/neutral) и 6 эмоций.

**Почему `blanchefort/rubert-base-cased-sentiment` для русского:**
Это RuBERT (BERT, дообученный на русскоязычном корпусе), fine-tuned именно на задаче sentiment в новостном/отзывном контексте. Альтернативы:
- `sberbank-ai/ruRoberta-large` — более мощная, но в 3× медленнее без значимого прироста точности на новостях
- ML-методы на bag-of-words (SVM, Naive Bayes) — быстро, но плохо на длинных текстах и саркастичных конструкциях

**Почему вероятности трёх классов, а не просто метка:**
Метка "нейтральный" у сообщения с распределением 45%/45%/10% и 5%/5%/90% — это разные вещи. Агрегатный `sentiment_score = positive_prob - negative_prob` даёт осмысленное среднее по теме: если 10 сообщений дали score -0.3 и 10 дали +0.1, среднее -0.1 честно отражает смещение в сторону негатива.

**Chunking для длинных текстов:**
BERT ограничен 512 токенами. Пост на 2000 слов разбивается на чанки с перекрытием 64 токена (чтобы не потерять контекст на границе) и усредняется. Без перекрытия концы и начала чанков анализируются без контекста — точность падает.

**Что пользователь поймёт из sentiment:**
- Всплеск negativity на графике sentiment dynamics → что-то тревожное происходит в этот час
- Канал с постоянно высоким `emotion_anger` → канал с агрессивной подачей материала
- Тема с `sentiment_shift` > 0.5 → тональность резко изменилась (новое развитие событий)

### 5. ner_extractor

**Что делает:** Извлекает именованные сущности (PERSON, ORG, GPE, LOC, PRODUCT и др.) и связи между ними.

**Почему Natasha для русского, а не spaCy или DeepPavlov:**
- spaCy для русского использует меньший корпус и хуже справляется с морфологически богатым русским (падежи, склонения). Натоша специально создана под русский NLP с включённым морфоанализатором — распознаёт "Путина", "Путиным", "Путину" как один объект.
- DeepPavlov — мощная альтернатива, но тяжелее в деплойменте (требует отдельного TF-сервера) и медленнее.
- Natasha = нативная Python-библиотека, нет внешних зависимостей, запускается в том же процессе.

**Почему BERT-NER для английского:**
`dslim/bert-base-uncased-ner` — стандартный CoNLL-2003 trained BERT, хорошо известное качество (F1 ~91% на CoNLL). Для английского это достаточно.

**Нормализация + WikiData:**
"Газпром", "ГАЗПРОМ", "Газпром Нефть" — без нормализации это три разных сущности в аналитике. Нормализованный текст используется как ключ при агрегации. WikiData ID позволяет в будущем обогащать профили сущностей внешними данными.

**Связи (entity_relations):**
```
Путин → назначил → Белоусов
Газпром → поставляет_в → Венгрия
```
Хранятся в `entity_relations` и строятся в граф Neo4j. Это основа для анализа: «какие организации часто упоминаются вместе с этой персоной?»

**Что пользователь поймёт:**
- Топ-10 персон за неделю → кто формирует повестку
- Граф co-occurrence → Газпром связан с Европой и энергетикой, а не с банковским сектором
- Spike упоминаний персоны → что-то важное произошло с этим человеком

### 6. topic_clusterer

**Что делает:** Группирует сообщения по темам без предварительной разметки.

**Алгоритм: Sentence Embeddings → UMAP → HDBSCAN**

Каждый шаг решает конкретную проблему:

**Sentence Transformers (`paraphrase-multilingual-mpnet-base-v2`)**
Превращает текст в вектор 768 измерений. Семантически похожие тексты («ЦБ поднял ставку» и «Банк России повысил ключевую ставку») будут близки в этом пространстве, хотя не имеют общих слов. Это принципиально важно для новостей, где одна тема выражается разными словами в разных каналах.

*Почему именно эта модель:* `paraphrase-multilingual-mpnet-base-v2` обучена на 50+ языках, включая русский. Это значит Russian embeddings и English embeddings находятся в одном векторном пространстве — можно кластеризовать смешанный контент без разделения по языку.

**Почему не TF-IDF + K-Means (классический подход):**
- TF-IDF не понимает синонимы — «ЦБ» и «Центробанк» как разные слова
- K-Means требует заранее указать K (количество тем). Сегодня их 15, завтра 40 — модель придётся перезапускать с новым K
- K-Means предполагает сферические кластеры одинакового размера — у новостей не так

**UMAP (Uniform Manifold Approximation):**
Уменьшает 768D → 2D, сохраняя локальную структуру. HDBSCAN эффективен в низкой размерности. Попытка запустить HDBSCAN напрямую в 768D неэффективна: алгоритм основан на расчёте расстояний, а в высокой размерности все точки оказываются почти равноудалены (проклятие размерности).

*Почему UMAP, а не PCA:* PCA — линейный метод, теряет нелинейные структуры. UMAP сохраняет кластерную структуру значительно лучше для текстовых эмбеддингов.

**HDBSCAN (Hierarchical DBSCAN):**
Находит плотные скопления точек. `min_cluster_size=5` — тема должна быть хотя бы в 5 сообщениях, иначе это шум. Метка `-1` (noise) для сообщений, не вошедших ни в одну тему.

*Почему не DBSCAN:* HDBSCAN устойчивее к выбору параметров, автоматически находит кластеры разной плотности. DBSCAN с одним `epsilon` не справляется, когда одна тема собрала 200 сообщений, а другая — 8.

**`public_cluster_id`** — детерминированный хеш, стабильный между перезапусками кластеризации. Позволяет фронтенду ссылаться на конкретную тему по постоянному URL.

**Периодичность:** Каждый час, окно 24 часа. Компромисс между актуальностью (можно было каждые 15 минут) и стоимостью (embeddings + UMAP + HDBSCAN на 10k сообщений занимает несколько минут).

### 7. topic_scorer

**Что делает:** Рассчитывает числовой показатель важности (`importance_score`) для каждой темы и присваивает уровень: `low / medium / high / critical`.

**Зачем нужен отдельный сервис, а не просто сортировка по количеству сообщений:**
Количество сообщений — плохой proxy для важности. Канал с 10 репостами одного поста создаёт большой кластер, но это не важная тема — это вирусный перепост. Тема с 20 сообщениями от 15 разных каналов, где половина сущностей — новые (novel), значительно важнее.

**9 признаков с весами:**

| Признак | Вес | Что измеряет | Почему важен |
|---------|-----|-------------|-------------|
| `growth_rate` | **0.22** | (recent - prev) / prev — скорость роста | Самый важный: активно растущая тема > статичная |
| `unique_channels` | **0.14** | log(кол-во каналов) | Тема в 10 каналах важнее, чем в одном |
| `message_count` | **0.12** | log(кол-во сообщений) | Объём — базовый сигнал |
| `unique_entities` | **0.10** | log(кол-во сущностей) | Богатая тема с многими акторами |
| `novelty` | **0.10** | Доля новых сущностей (не виданных за 14 дней) | Признак реально нового события |
| `sentiment_intensity` | **0.10** | max(\|avg_sentiment\|, negative_share) | Эмоциональные темы привлекают внимание |
| `new_channel_ratio` | **0.08** | Доля каналов, которые впервые пишут об этом | Тема распространяется, захватывает новые площадки |
| `cluster_density` | **0.08** | Плотность графа сущностей | Тема с плотными связями между акторами более конкретна |
| `sentiment_shift` | **0.06** | \|recent_sentiment - prev_sentiment\| | Резкая смена тональности = что-то произошло |

**Почему log-scale для message_count и unique_channels:**
Без log разница между 100 и 200 сообщениями была бы такой же, как между 1000 и 1100. С log(x) прирост на первые 100 сообщений важнее, чем следующие 100 — это соответствует тому, как люди воспринимают значимость.

**Нормализация min-max в рамках текущего запуска:**
Каждый признак нормализуется относительно других тем в том же запуске кластеризации. Это означает, что score = 0.8 говорит не «в абсолютном смысле это важная тема», а «из всех тем сегодня эта находится в топ-20%». Это устойчиво к изменению общей активности (выходные vs будни).

**Small cluster penalty:**
Если в теме < 3 сообщений, итоговый score умножается на 0.5. Нельзя быть "critical" с 2 сообщениями, даже если они от разных каналов с высокой скоростью роста — слишком мало данных для уверенного вывода.

**Уровни важности:**
- `critical` ≥ 0.85 — событие, которое явно доминирует в повестке
- `high` ≥ 0.65 — значимая тема, требует внимания
- `medium` ≥ 0.35 — рядовая новость
- `low` < 0.35 — единичные упоминания

**Режимы запуска:**
- `batch` — переоцениваем все темы последнего запуска кластеризации
- `scheduled` — автоматически каждые 5 минут
- `oneshot` — оцениваем одну конкретную тему по запросу (например, при создании нового кластера)

**Что пользователь поймёт:**
- Лента тем, отсортированная по `importance_score`, показывает самое важное вверху — не просто самое активное
- Тема с `novelty=0.9` и `new_channel_ratio=0.8` = что-то действительно новое, о чём ещё мало кто знал
- Тема с высоким `sentiment_shift` при умеренном `growth_rate` = неожиданный поворот в устоявшейся теме

### 8. source_resolver

**Что делает:** Определяет первоисточник каждой темы — какой канал/сообщение впервые написал об этом.

**Алгоритм:**
1. Берёт все сообщения кластера, сортирует по дате (`message_date ASC`)
2. Ищет совпадения по `primary_url_fingerprint` — если несколько сообщений содержат одну ссылку, первое хронологически = источник
3. Если нет URL — сравнивает множества нормализованных сущностей: максимальный Jaccard overlap с самым ранним сообщением = источник
4. Проверяет `forward_from_channel` / `reply_to_message_id` — прямые форварды дают однозначный ответ

**Два типа разрешения:**
- `exact` — прямой форвард или идентичный URL → высокая уверенность
- `inferred` — вывод по entity overlap или temporal proximity → средняя уверенность

**Зачем хранить оба типа раздельно:**
`exact` неизменяем — форвард есть форвард. `inferred` может обновиться при появлении новых сообщений в кластере. Раздельное хранение позволяет перезапускать inferred-резолюцию без потери exact-результатов.

**Что пользователь поймёт:**
- Какой канал формирует повестку (является первоисточником чаще всего)
- Какие каналы только ретранслируют (высокий процент inferred-форвардов)
- Время от первого упоминания до пика распространения

### 9. analytics_api

**Что делает:** Read-only REST API, единственная точка доступа фронтенда к данным.

**Почему read-only и отдельный сервис:**
Аналитические запросы требуют сложных JOIN'ов через несколько таблиц. Если бы каждый ML-сервис сам отвечал на вопросы UI, логика агрегации дублировалась бы. Единый API централизует её и защищает production-базу от сложных запросов напрямую из браузера.

**Ключевые вычисления:**

`signed_sentiment_score`:
```sql
COALESCE(positive_prob, 0) - COALESCE(negative_prob, 0)
```
Диапазон [-1, +1]. Почему не просто `sentiment_score` из модели? Модель возвращает score относительно предсказанного класса (т.е. confidence). Если предсказан "negative" с уверенностью 0.9, `sentiment_score=0.9`, но нет информации насколько это "negative". `positive_prob - negative_prob` — это честная мера смещения.

**Topic Evolution (topic_evolution.py):**
Временно́й анализ развития темы — бакеты 15m / 1h / 1d с обнаружением событий:
- `spike` — резкий рост количества сообщений
- `channel_expansion` — тема захватила новые каналы
- `sentiment_shift` — тональность изменилась

Это позволяет LLM_enricher знать не просто «что в теме», но «как тема эволюционировала».

**Кеширование (in-memory, 900s TTL):**
Граф-аналитика (PageRank, bridge nodes) вычисляется локально в Python на каждый запрос. 900 секунд — достаточно долго, чтобы не вычислять для каждого пользователя, но коротко, чтобы граф обновлялся при появлении новых данных.

**Интеграция с llm_enricher:**
API проксирует запросы к llm_enricher через HTTP. Если enricher недоступен — возвращает `null` для LLM-полей, а не 500.

### 10. llm_enricher

**Что делает:** По запросу генерирует текстовые объяснения тем через Mistral API: краткое резюме, ключевые инсайты, объяснение новизны, метку темы.

**Типы enrichment:**

| Тип | Что возвращает |
|-----|---------------|
| `summary` | 2-3 предложения: суть темы простым языком |
| `explanation` | Детальный анализ: кто, что, почему важно |
| `novelty_explanation` | Объяснение почему тема новая, что в ней необычного |
| `label` | Короткая метка темы (5-7 слов) для отображения |

**Почему Mistral, а не GPT-4 или локальная модель:**
- Mistral Large — сильная модель с хорошим качеством для русского языка
- Цена: $2/Mtok input, $6/Mtok output — в 5-10× дешевле GPT-4
- Локальная модель (Llama 3, Mistral 7B) потребовала бы GPU, которого нет в инфраструктуре
- Провайдер абстрагирован: `LLMProvider` interface позволяет сменить Mistral на другого без изменений в остальном коде

**Контекст, передаваемый в LLM:**
```python
ClusterEnrichmentInput:
  - representative_messages (топ-5 сообщений)
  - top_entities (PERSON, ORG с количеством упоминаний)
  - sentiment_summary (positive/negative/neutral распределение + dominant_emotion)
  - timeline_buckets (как тема развивалась по времени)
  - evolution_events (обнаруженные аномалии: spike, sentiment_shift)
  - graph_top_nodes (PageRank-лидеры графа сущностей)
  - importance_score + breakdown (зачем тема вообще важна)
```
Чем богаче контекст — тем точнее объяснение. LLM не «придумывает» факты — всё основано на данных из базы.

**Бюджет $5/день:**
При `mistral-large-latest` ~500-700 токенов input + ~300-500 токенов output ≈ $0.002 на вызов. $5 = ~2500 enrichments в день. Бюджет отслеживается в таблице `llm_budget_tracking` с учётом реальной стоимости каждого вызова.

**Circuit Breaker:**
Если Mistral API не отвечает 5 раз подряд, circuit breaker открывается на 30 секунд — сервис перестаёт пытаться и возвращает кеш. Это предотвращает накопление очереди ожидающих запросов при временном сбое провайдера.

**TTL кеша: 7 дней:**
Объяснение темы редко меняется принципиально после того, как тема сформировалась. 7 дней — разумный баланс между актуальностью и стоимостью.

### 11. auth_service

**Что делает:** FastAPI-сервис для аутентификации, управления пользователями, реакций на сообщения.

**Почему отдельный сервис, а не middleware в analytics_api:**
Auth — это отдельная зона ответственности с собственной базой данных. Если завтра нужно добавить SSO или OAuth — это делается в auth_service без изменений в analytics_api. Отдельная БД (`tg_news_auth`) изолирует токены и пароли от аналитических данных.

**JWT access + refresh token:**
- Access token — короткий (30 минут), stateless: любой сервис может проверить подпись без обращения к БД
- Refresh token — длинный (30 дней), хранится как хеш в `refresh_sessions`: при logout инвалидируется немедленно
- Это стандартная схема OAuth2. Access token отзвать нельзя до истечения — это компромисс между производительностью (нет lookup в БД) и безопасностью

**Argon2 для паролей:**
Argon2id — победитель Password Hashing Competition 2015. Устойчив к GPU-атакам благодаря memory-hard алгоритму. bcrypt и scrypt — достаточные альтернативы, но Argon2 является текущим стандартом.

**User reactions (like/dislike/flag):**
`flag` используется для краудсорсинговой модерации. Помеченные сообщения видны администраторам в аудит-логе. Это позволяет пользователям помечать дезинформацию.

### Neo4j: граф сущностей

**Узлы:** `Message`, `Entity`, `Channel`, `Topic`

**Рёбра:**
- `MENTIONED_IN` (Entity → Message) — сущность упомянута в сообщении
- `CO_OCCURS_WITH` (Entity ↔ Entity) — сущности упоминаются в одном сообщении (с весом = количество со-упоминаний)
- `POSTED_IN` (Message → Channel)
- `BELONGS_TO` (Message → Topic)

**Почему PageRank для graph_top_nodes:**
PageRank в графе co-occurrence находит "хабы" — сущности, связанные с максимальным числом других сущностей. Это не то же самое, что частота упоминаний: ЦБ РФ может упоминаться реже Путина, но быть связан с большим числом других акторов (банки, курс рубля, инфляция).

## Метрики и наблюдаемость

### Что каждая метрика говорит

**`kafka_consumer_lag{service="sentiment_analyzer"} > 10000`**
→ Сервис не справляется с потоком. GPU-инференс слишком медленный или batch_size слишком мал. Решение: увеличить `batch_size` или добавить реплику.

**`kafka_messages_dlq_total / kafka_messages_processed_total > 0.01`**
→ DLQ rate > 1%. Что-то системно не так: схема поменялась, модель возвращает неожиданный формат, база недоступна. Нужно читать DLQ-сообщения вручную.

**`analytics_api_request_latency_seconds{route="/analytics/clusters",quantile="0.99"} > 2`**
→ 1% запросов к списку тем занимает > 2 секунд. Нужен индекс на `cluster_assignments(run_id, public_cluster_id)` или materialized view.

**`SCORED_TOPICS_TOTAL{level="critical"} > 5 в день`**
→ Много "critical" тем — либо действительно турбулентный день, либо веса в topic_scorer требуют калибровки.

**`CLUSTERING_DURATION > 300s`**
→ UMAP/HDBSCAN на текущем объёме данных занимает > 5 минут. Нужно уменьшить окно (`window_hours`) или увеличить `min_cluster_size`.

**`ENRICH_LATENCY{type="summary"} > 10s`**
→ Mistral API медленно отвечает. Circuit breaker сработает после 5 подряд медленных ответов.

**`COST_USD` в llm_enricher**
→ Накопительный счётчик расходов. При приближении к daily_usd budget — автоматически перестаём делать новые вызовы.

### Observability UIs

- **Kafka UI** http://localhost:8080 — визуальный мониторинг топиков, consumer lag, DLQ
- **Neo4j Browser** http://localhost:7474 — Cypher запросы к графу
- **Prometheus** http://localhost:9090 — все метрики сервисов
- **Grafana** http://localhost:3001 — дашборды (профиль: `monitoring`)

### Unit-тесты (tests/unit/)

Каждый unit-тест тестирует **чистую функцию** без I/O: никаких DB-коннектов, никаких Kafka-продюсеров. Это возможно, потому что весь критический бизнес-код намеренно изолирован от инфраструктуры — `features.py`, `scoring.py`, `resolution.py`, `text_processing.py` — всё pure Python.

#### test_topic_scorer_scoring.py — корректность формулы скоринга

Самый плотный тест-файл проекта. Проверяет каждое свойство scoring-алгоритма отдельно:

| Класс теста | Что проверяет |
|-------------|--------------|
| `TestScoringMonotonicity` | Монотонность: больше сообщений → выше score; выше growth_rate → выше score; больше каналов → выше score |
| `TestScoringBounds` | Score всегда в [0.0, 1.0]; минимальный кластер не ломает функцию; **сумма всех весов = 1.0** |
| `TestSmallClusterPenalty` | Penalty применяется при `message_count < 3`; breakdown содержит описание штрафа; при `message_count == threshold` штрафа нет |
| `TestLevelThresholds` | При norm_val=1.0 → `critical`; при norm_val=0.0 → `low`; проверка кастомных порогов |
| `TestBreakdownJsonSchema` | breakdown содержит все 9 компонент; сумма contributions = raw_weighted_sum; версия скоринга прокидывается |
| `TestConfigurableWeights` | При weight=0.0 вклад компонента = 0.0; формула не зависит от нуля |

**Ключевой паттерн — попарная нормализация `_score_pair`:**
Поскольку нормализация min-max вычисляется в рамках одного запуска, тест для проверки монотонности создаёт два кластера одновременно, нормализует их вместе и сравнивает результаты. Одиночный кластер всегда получит 0.5 по всем признакам — это не баг, а следствие нормализации.

#### test_topic_scorer_features.py — чистая математика признаков

Тестирует `compute_raw_features` в изоляции:
- `growth_rate` при `prev=0` клипируется до 5.0 (защита от деления на ноль)
- `growth_rate` при падении клипируется до -1.0
- `message_count` логарифмируется через `log1p` — при `message_count=0` возвращает 0.0
- `new_channel_ratio = new_channel_count / unique_channels` — правильное деление

#### test_topic_scorer_regression.py — золотые фикстуры

**Самый важный тест для стабильности.** 5 именованных кластеров с заранее известными характеристиками, чьи exact score-значения зафиксированы в коде:

| Кластер | Характеристика | Ожидаемый уровень |
|---------|---------------|------------------|
| `viral_breaking` | 80 сообщений, 15 каналов, growth=7x, 18/20 новых сущностей | `critical` или `high` |
| `slow_background` | 6 сообщений, стабильный рост, нейтральный sentiment | `low` или `medium` |
| `tiny_cluster` | 1 сообщение, 0 сущностей | `low` (с penalty) |
| `sentiment_spike` | умеренный объём, резкий sentiment_shift | выше `slow_background` |
| *(5-й кластер)* | *(mixed signals)* | проверка граничного случая |

**Зачем нужны регрессионные тесты:** Если кто-то меняет веса в `config.py` или формулу в `features.py` без обновления `scoring_version` — тест упадёт. Это предотвращает тихое изменение алгоритма, которое изменит ранжирование тем без явного намерения.

#### test_analysis_contracts.py — контракты между сервисами

Проверяет, что **события, которые один сервис публикует в Kafka, соответствуют JSON Schema**, которую ожидает следующий сервис:

- NER co-occurrence relations дедуплицируются: `(ЦБ, Банк России)` — одна и та же normalized сущность `Центральный Банк`, поэтому со-упоминание с `Москва` добавляется только один раз
- `topic_assignment` событие от topic_clusterer валидируется против `schemas/topic_assignment.schema.json`
- Многоязычный preprocessing event (`analysis_mode="partial"`, `language="other"`) валидируется против `schemas/preprocessed_message.schema.json`
- Fallback кластеризация (< min_cluster_size сообщений) создаёт singleton-кластеры, а не падает
- `run_id` детерминирован: одинаковый набор `event_id` в любом порядке → одинаковый `run_id`

**Почему это важно:** JSON Schema — это единственный «контракт» между сервисами. Если preprocessor добавит новое обязательное поле, которого нет в схеме sentiment_analyzer — это обнаружат эти тесты до деплоя, а не в проде.

#### test_preprocessor_fingerprints.py — детерминизм обработки текста

- URL нормализация: `utm_source=tg`, `utm_campaign=x`, `#fragment` удаляются; параметры сортируются — одна страница = один fingerprint
- Текст с Telegram-шаблонами (`Подписывайтесь на @banksta`, `Канал РБК в MAX`, `https://t.me/rbc_news`) после очистки → только содержательный текст
- `simhash64` и `normalized_text_hash` всегда непустые
- `primary_url_fingerprint == url_fingerprints[0]` (первый URL = главный)
- Язык `ru` и `en` → `analysis_mode="full"`, остальные → `"partial"`

#### test_source_resolution.py — алгоритм определения первоисточника

- **Quote match:** сообщение, цитирующее фрагмент ≥ 20 символов из более раннего сообщения, получает `source_type="quoted"` и confidence > 0.5
- **Fallback:** при отсутствии evidence → `source_type="earliest_in_cluster"`, confidence=0.35 (намеренно низкая — это признание неопределённости)
- **Unknown:** если вообще нет данных → `source_type="unknown"` с явным explanation

#### test_graph_analytics.py — анализ графа сущностей

- `bank` упоминается в двух разных сообщениях (events `a:1` и `b:2`) → при построении графа она становится bridge node (соединяет два разных документа-контекста)
- PageRank выводит `bank` на первое место в `top_entities`
- Пустой граф → `is_small_graph=True`, пустые `top_entities`, без падения

#### test_llm_enricher_budget.py — финансовый контроль LLM

- Запрос за $0.01 при бюджете $1.0 → разрешён
- Запрос за $0.01 при бюджете $0.001 → отклонён немедленно
- Два запроса по $0.03 при бюджете $0.05 → второй отклонён (накопительный учёт)
- `warm_start(pool)` загружает уже потраченное из БД — при перезапуске сервиса бюджет не сбрасывается
- `estimate_cost(1M input, 0.5M output)` = $2 + $3 = $5.0 (по тарифу $2/$6 за Mtok)
- `record_actual` корректирует расход: зарезервировал $0.10, реально потрачено $0.07 → счётчик = $0.07

#### test_llm_enricher_schema_validation.py — валидация LLM-ответов

LLM может вернуть что угодно. Поэтому каждый тип обогащения имеет JSON Schema и тест на ней:
- `cluster_label` требует поля `label` (≤ 80 символов) и `confidence` в [0.0, 1.0]
- `cluster_summary` требует `summary` + `key_points` (не более 5 пунктов)
- `novelty_explanation` требует `novelty_verdict` + `rationale`
- Отсутствие обязательного поля → `ValidationError`; уверенность 1.5 → `ValidationError`

#### test_llm_enricher_language_routing.py

Параметризованный тест: `ru` → `"full"`, `en` → `"full"`, `de/zh/ar/fr` → `"partial"`, `None/"und"/""` → `"unknown"`. Проверяет, что маршрутизация к разным промптам (русский / английский / другой) работает правильно.

#### test_topic_comparison.py — классификация пар тем

| Сценарий | Ожидаемый результат |
|---------|---------------------|
| Те же сущности, каналы, общий message_hash | `same_topic`, similarity ≥ 0.74 |
| Пересечение сущностей, но разные каналы + sentiment расходится | `possible_subtopic_split` |
| Нет пересечения сущностей, разные каналы, далеко по времени | `different_topics` |

#### test_topic_evolution.py — обнаружение событий в жизни темы

- Bucket floor: `10:37:45` → `10:30` для 15m, `10:00` для 1h, `00:00` для 1d
- Невалидный bucket `"5m"` → `ValueError`
- 1 сообщение в 1 bucket → 1 точка, событие `topic_created`
- 2 сообщения в 10:00, 5 в 11:00 (другой канал, резкий sentiment_shift) → обнаруживается `channel_expansion` или `sentiment_shift` событие

#### test_collector_events.py — маппинг CollectedMessage → Kafka event

Проверяет, что все 17 полей (включая `grouped_id`, `reply_to_message_id`, `forward_origin_type`, `media.type`) правильно попадают в event-конверт. Тест существует, потому что маппинг — это граница между Telegram API и нашим форматом; любая опечатка в имени поля приведёт к тихой потере данных.

### Integration-тесты (tests/integration/)

Все требуют `TEST_DATABASE_DSN` и **применённых миграций**. Каждый тест создаёт изолированную схему (`test_scorer_{uuid}`) и удаляет её в `tearDown` — тесты не загрязняют друг друга.

#### test_topic_scorer_pg.py — полный round-trip скоринга

1. Создаёт тестовую схему
2. Сидирует: `cluster_runs_pg`, `raw_messages`, `cluster_assignments` (6 сообщений, 2 кластера)
3. Добавляет `sentiment_results` и `ner_results` для части сообщений
4. Запускает `TopicScorerService.run_batch(run_id)` против реальной БД
5. Проверяет: записи появились в `topic_scores`, score в [0.0, 1.0], уровень валиден, breakdown содержит все 9 компонент, версия = `"v1"`

**Почему важен этот интеграционный тест, если есть unit-тесты:** Unit-тесты проверяют формулу. Этот тест проверяет, что SQL-запросы в `repository.py` правильно собирают признаки из реальной схемы БД. Если миграция 009 не добавила нужный столбец — unit-тест пройдёт, интеграционный — упадёт.

#### test_analysis_pipeline_pg.py — полный пайплайн анализа

Запускает три сервиса последовательно против одной БД:
1. `SentimentAnalyzerService` — обрабатывает тестовое сообщение, пишет в `sentiment_results`
2. `NerExtractorService` — пишет в `ner_results` и `entity_relations`
3. `TopicClustererService` — кластеризует, пишет в `cluster_assignments`

Проверяет, что:
- Все три записи существуют в БД после прогона
- `trace_id` сквозной — один и тот же UUID во всех трёх таблицах
- Kafka-события соответствуют схемам (через `FakeProducer` — мок, который перехватывает события без реального Kafka)

**`FakeProducer`** — намеренно минималистичный мок: `send_and_wait(topic, value, key, headers)` просто декодирует JSON и добавляет в список. Это достаточно для проверки того, что сервис _пытается_ опубликовать правильное событие, без необходимости запускать Kafka.

#### test_first_source_pg.py — source resolution против реальной БД

Сидирует кластер из 3 сообщений: одно с URL fingerprint, два без. Запускает `SourceResolverService`. Проверяет, что:
- Запись появилась в `cluster_source_resolutions`
- Сообщение с URL fingerprint определено как источник (exact resolution)
- Confidence у exact ≥ 0.7

#### test_llm_enricher_pg.py — кеш и бюджет в реальной БД

- Первый запрос → `cache_hit=False`, результат записан в `llm_enrichments`
- Второй запрос с тем же `cluster_id+enrichment_type` → `cache_hit=True`, LLM не вызывался
- Расходы записаны в `llm_budget_tracking`
- Использует `MockProvider` — не делает реальных HTTP-запросов к Mistral

#### test_topic_comparison_pg.py + test_topic_evolution_pg.py

Те же логические проверки, что в unit-тестах, но SQL-запросы выполняются против реальной схемы. Гарантируют, что `JOIN`'ы и `GROUP BY` в API-сервисе возвращают корректные данные при разных конфигурациях данных.

### Frontend-тесты (Jest + React Testing Library)

#### frontend/src/__tests__/utils.test.ts

Тестирует утилиты форматирования и цветовой семантики:

| Функция | Что тестируется |
|---------|----------------|
| `formatNumber` | 0-999 → без суффикса; 1000 → `"1.0K"`; 1_000_000 → `"1.0M"` |
| `sentimentColor` | score > 0.2 → `var(--positive)`; < -0.2 → `var(--negative)`; между → `var(--neutral-sentiment)` |
| `sentimentLabel` | 0.5 → `"Positive"`; -0.5 → `"Negative"`; 0 → `"Neutral"` |
| `entityTypeColor` | `"PER"` → `var(--entity-per)`; регистронезависимо; неизвестный тип → `var(--entity-misc)` |

**Почему тестируются пороги sentiment:** В UI цвет "neutral" применяется к диапазону [-0.2, 0.2], а не только к 0.0. Тест фиксирует эти границы — если дизайнер захочет сузить диапазон до [-0.1, 0.1], тест явно покажет, что контракт изменился.

#### frontend/src/__tests__/auth.test.ts

Тестирует управление токенами в `localStorage`:
- При пустом хранилище → `null` для обоих токенов
- После `storeTokens` → корректное чтение
- После `clearTokens` → снова `null`

Использует ручной мок `localStorage` (не jsdom), потому что нужен полный контроль над состоянием между тестами.

#### frontend/src/__tests__/source-ui.test.tsx + sources-page.test.tsx

Рендерят React-компоненты с моками API-хуков, проверяют, что UI-элементы (кнопки видимости канала, список источников) рендерятся и реагируют на события.

### Что не тестируется (намеренно)

| Область | Почему нет тестов |
|---------|------------------|
| ML-модели (RuBERT, Natasha, UMAP/HDBSCAN) | Качество моделей не меняется от кода; проверяется вручную на реальных данных |
| Telegram API (Telethon) | Внешний сервис; мокирование хрупко и не даёт ценности |
| Kafka продюсеры/консьюмеры | Покрыты `FakeProducer` в integration-тестах; реальная Kafka — это e2e |
| Neo4j граф | Нет integration-тестов для Neo4j; граф — производная от NER, которое тестируется |
| Grafana/Prometheus | Инфраструктура, а не бизнес-логика |
